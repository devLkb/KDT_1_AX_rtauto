## 0. 전체 그림 — 왜 이 순서인가

ML-Agents 학습이 성립하려면 다음 4가지가 **전부** 갖춰져야 한다:

1. **믿을 수 있는 물리** — 정책이 상호작용할 시뮬레이션이 진동·발산 없이, 그리고 **가속 배속에서도** 안정.
2. **학습할 태스크의 정의** — 관측(Observation)·행동(Action)·보상(Reward)·에피소드(Reset)가 명확한 환경.
3. **(IL의 경우) 양질의 시연 데이터** — 텔레오퍼레이션이 Agent의 행동 경로를 그대로 통과하며 기록된 데모.
4. **학습 인프라** — ML-Agents 패키지/Python 버전 호환, 가속·병렬 실행, 평가 체계.

현재 프로젝트는 1번의 실시간(1배속) 안정성과 3번의 원료(텔레옵 파이프라인)를 확보한 상태다.
나머지는 아직 시작 전이며, 특히 **"태스크 정의"와 "행동 공간 확정"은 데모를 한 번 녹화하면
바꿀 수 없으므로**(스펙이 다르면 데모 전체 무효) 가장 신중하게 결정해야 한다.

```
[현재] 디지털 트윈 + 텔레옵(손 9DoF)
   │
   ├─ Phase 5  텔레옵 트랙 마무리 (검증 런 → rate limit → [태스크에 따라] 팔 IK)
   ├─ Phase 6  ML-Agents 설치 + 스모크 테스트 (예제 1회 학습)
   ├─ Phase 7  물리 RL-readiness (물체+중력, 접촉, 배속, 리셋)
   ├─ Phase 8  태스크·Agent 설계 (obs/action/reward 확정) ← 여기서 스펙 동결
   ├─ Phase 9  모방학습 데모 수집 (Heuristic 재배선 → 시연 녹화)
   └─ Phase 10 학습 실행 (BC 사전학습 → GAIL+PPO → 평가·반복)
```

## 2. Phase 5 — 텔레옵 트랙 마무리 (IL 데모 품질의 전제)

모방학습의 데모는 텔레옵으로 만든다. **텔레옵의 노이즈·지연·미도달이 그대로 정책의 상한이 되므로**,
데모 수집 전에 텔레옵 품질을 끌어올려 놓아야 한다. "쓰레기 시연 → 쓰레기 정책"은 IL의 기본 법칙.

### 2-1. 비전 검증 런 (진행 중 — 남은 관문)
- [ ] 주먹 때 검지·중지 **근위 도달률 ≥ 80%** (트림 전 25/29% → 트림 후 확인)
- [ ] thmOpp 주먹 때 SVH값 ~0.99 (스왑 방향 확인)
- [ ] spread 상수 0 고정 상태에서 주먹 시 벌어짐 없는지 확인
- 도구: `vision_node.py` 실행 → 로그 CSV → `analyze_tracking.py`

### 2-2. Phase 4 안전장치 (데모 수집 전 완료 권장)
- [ ] **rate limit**: 프레임 간 |Δangle| 상한(관절별). 비전 스파이크가 급명령이 되는 것 차단.
- [ ] **One Euro 파라미터 튜닝**: min_cutoff/beta — 지연 vs 떨림 트레이드오프 확정.
- [ ] **spread 게이팅**(현재 상수 0 임시 고정): 굽힘 크면 spread 0으로 누르는 게이팅으로 대체할지 결정.
- [ ] (권장) 워치독: 일정 시간 미검출 시 안전자세 복귀.

**왜 필수인가**: IL은 (관측, 행동) 쌍을 그대로 흉내 낸다. 시연에 지터·스파이크가 섞이면
정책이 그 노이즈까지 학습한다. GAIL은 특히 시연 분포를 판별자로 흉내 내므로 민감하다.

### 2-3. 팔(UR5e) 텔레옵 — 태스크에 따라 필수/생략
- 현재 텔레옵은 **손 9 DoF만** 제어. 팔은 고정 포즈.
- **태스크가 "손 안 조작"(in-hand)이나 "고정 위치 파지"면**: 팔 텔레옵 없이 진행 가능. 가장 빠른 경로.
- **태스크가 "집어서 옮기기(pick-and-place)"면**: 시연에 팔 움직임이 필요 →
  MediaPipe 손목 pose(위치+방향) → UR5e IK(6축 해석해 또는 Unity에서 CCD/Jacobian 반복해) 구현 필요.
  단일 카메라는 z(깊이)가 부정확하므로 **팔 시연의 품질은 스테레오(5단계) 없이는 낮다**는 점 유의.
- **권고**: 첫 IL/RL 사이클은 팔 고정 + "물체를 손 앞에 스폰 → 파지" 태스크로 시작해
  팔 IK·스테레오는 태스크를 확장할 때 붙인다. (뒷단 재사용 원칙과 동일한 전략)

## 4. Phase 7 — 물리의 RL-readiness (조건이 가장 많은 구간)

지금 씬은 "텔레옵 디지털 트윈"에 최적화되어 있다. RL 환경은 요구가 다르다.
**아래 4개를 전부 통과해야 학습을 시작할 수 있다.**

### 4-1. 상호작용 물체 + 중력 도입
현재: 모든 ArticulationBody `useGravity=false`, 씬에 물체 없음.
- [ ] 파지 대상 물체(Rigidbody) 추가: 원기둥/큐브/구 등 단순 프리미티브부터. 질량 0.05~0.3kg,
  적절한 PhysicMaterial(마찰 dynamicFriction/staticFriction ≥ 0.6, bounciness 0).
- [ ] **물체에는 중력 ON**. 로봇은 gravity off 유지 가능(위치 드라이브가 포즈를 유지하므로) —
  단, 실물 이전(sim-to-real)까지 갈 계획이면 로봇도 중력 켜고 드라이브가 버티는지 별도 확인.
- [ ] `SvhSelfCollisionIgnore`는 **로봇 내부 쌍만** 무시하므로 물체와의 충돌은 살아있음 — 코드 확인 완료
  상태이나, 물체 추가 후 실제로 손가락-물체 접촉이 발생하는지 1회 검증.

### 4-2. 접촉(파지) 안정성 프로브 — 합격 조건
드라이브가 stiffness 10000 / forceLimit 100000으로 매우 강하다. 물체와 접촉하면
**관통·떨림·물체 튕겨나감**이 새로 발생할 수 있다 (7/2~7/8 진동 사건은 전부 무접촉 상태에서의 검증).
- [ ] **파지 프로브**: 물체를 손바닥 앞에 놓고 주먹 명령 → ① 물체가 튕겨나가지 않음
  ② 손가락이 물체를 관통하지 않음(침투 깊이 < 2~3mm) ③ 쥔 상태 10초 유지 시 물체 떨림 σ < 1mm.
- [ ] 실패 시 조정 카드(순서대로): 접촉 오프셋(contactOffset), fixedTimestep 0.02→0.01,
  물체 질량 상향, 손가락 드라이브 forceLimit 하향(실물 SVH 토크에 근사시키면 sim-to-real에도 유리),
  solverIterations 상향.
- 계측: `SvhJointLogger` 재활용 + 물체 pose 로거 추가.

### 4-3. 가속 배속 안정성 — **RL의 숨은 필수 조건**
`mlagents-learn`은 기본 **`--time-scale=20`** 으로 돌린다. 1배속에서 안정해도 배속에서 깨질 수 있다.
- [ ] Time.timeScale = 10, 20에서 §4-2 파지 프로브 + 주먹↔펴기 프로브 재실행 →
  1배속과 동일 판정(리밋 침범 0%, 잔여진동 0, 파지 유지) 통과해야 함.
- [ ] 실패 시: 학습 설정에서 `--time-scale`을 낮추거나(학습 느려짐) fixedTimestep 축소.
- ⚠️ 이 프로젝트는 초경량 링크(1e-6 kg·m²) + 고강성 드라이브 조합이라 **배속 취약 후보** —
  반드시 실측으로 확인하고 넘어갈 것. (7/7~7/8 진동 사건의 교훈: 가정하지 말고 프로브로 확정)

### 4-4. 에피소드 리셋의 완전성 — 합격 조건
RL은 에피소드를 수천~수만 번 리셋한다. 리셋이 1%만 오염돼도 학습 데이터가 오염된다.
- [ ] 리셋 루틴 구현: 전 관절 `jointPosition`/`jointVelocity` 직접 세팅(텔레포트) + `xDrive.target` 동기화
  + 물체 재스폰(위치 랜덤화 포함). `SvhInitialPoseSync.cs`가 이미 같은 원리 — 일반화하면 됨.
- [ ] **연속 리셋 프로브**: 100회 연속 리셋 → 매회 리셋 직후 0.5초간 전 관절 속도 |ω| < 0.01 rad/s,
  포즈 오차 < 0.5°. "초기 킥"(문제 ⑭)이 리셋마다 재발하지 않는지 확인.
- [ ] 물체 스폰 랜덤화 범위 정의(예: 손바닥 앞 5×5×5cm 박스 내 균등분포) — 도메인 랜덤화의 최소 단위.

## 5. Phase 8 — 태스크·Agent 설계 (스펙 동결 지점)

### 5-1. 태스크 결정 — 사용자가 정해야 할 첫 번째 사항
| 후보 | 팔 필요 | 난이도 | 비고 |
|------|---------|--------|------|
| **A. 고정 팔 파지(grasp)**: 손 앞 랜덤 위치 물체를 쥐고 3초 유지 | ✕ | ★★ | **첫 사이클 권장.** 손 9 DoF만, 현 텔레옵으로 시연 가능 |
| B. 파지+들어올리기(lift): 쥔 물체를 10cm 들어올림 | 팔 1~2축 or 손목만 | ★★★ | A 성공 후 커리큘럼 확장 |
| C. Pick-and-place: 집어서 목표 지점에 놓기 | ○ (6축+IK) | ★★★★★ | 팔 텔레옵(§2-3)+스테레오 선행 필요 |

### 5-2. Agent 설계 — 확정해야 할 스펙 (데모 녹화 전 동결)
**Observation (예: 태스크 A 기준, 정규화 필수)**
- 손 9관절 위치 + 속도 (18) — `jointPosition`/`jointVelocity`
- 손가락 끝 5개의 손바닥 기준 상대 위치 (15)
- 물체의 손바닥 기준 상대 pose (7: 위치3+쿼터니언4) + 속도 (3)
- 합계 ~43 float. (카메라 관측은 첫 사이클에선 쓰지 않음 — 상태 관측이 수렴 훨씬 빠름)

**Action — 가장 중요한 결정**
- 권장: **연속 9차원, 현재 target 대비 Δtarget** (스케일: 관절 가동범위의 ±5%/step).
  절대각보다 Δ방식이 정책이 부드럽고 rate limit과 개념이 일치.
- ⚠️ **여기서 정한 차원·의미·스케일이 바뀌면 녹화한 데모 전량 폐기.** 신중히 정하고 문서화.

**Decision 주기**
- `DecisionRequester` 주기 5 (fixedTimestep 0.02 기준 10Hz) 권장 — 비전 전송률(SEND_HZ_CAP)과
  같은 자리수로 맞추면 시연-정책 간 시간 스케일이 일치.

**Reward (RL용, 태스크 A 예)**
- 매 스텝: `-0.001`(시간 페널티) + `0.1 × (이전 스텝 대비 손바닥-물체 거리 감소)`
- 접촉 보너스: 손가락 2개 이상 + 엄지가 물체 접촉 시 `+0.01/step`
- 성공: 파지 유지 3초 → `+1.0` + 에피소드 종료 / 물체 이탈(낙하·튕김) → `-0.5` + 종료
- max step: 500 (10초 @ 50Hz 물리, decision 10Hz면 100 decision)

### 5-3. 산출물
- [ ] `docs/AGENT_SPEC.md` — obs/action/reward/reset 스펙 표 + 버전 번호. 데모·모델·config가 전부
  이 버전을 참조하게 함.

## 6. Phase 9 — 모방학습(IL) 데모 수집

### 6-1. 아키텍처 조건 (핵심): 텔레옵을 Agent의 Heuristic으로 재배선
**ML-Agents의 데모는 "(Agent의 관측, Agent의 행동)" 쌍이다.** 현재처럼 `SvhHandDriver`가
`HandSliderUI`를 거쳐 xDrive에 직접 주입하면 ML-Agents는 아무것도 기록할 수 없다.

```
[현재]  UDP → SvhReceiver → SvhHandDriver → HandSliderUI → xDrive
[필요]  UDP → SvhReceiver → Agent.Heuristic()이 action으로 반환
            → OnActionReceived(action) → xDrive     ← 정책 실행과 완전히 같은 경로
            → DemonstrationRecorder가 (obs, action) 자동 기록
```
- [ ] Agent 클래스의 `Heuristic()`에서 UDP 최신값을 §5-2에서 정한 action 형식(Δtarget)으로 변환해 반환.
  (절대각 수신값 → Δ 변환 로직 필요: `Δ = clamp(수신각 - 현재 target, ±상한)`)
- [ ] `OnActionReceived()`가 시연·학습·추론 모두에서 **유일한 구동 경로**가 되도록 기존
  `SvhHandDriver` 직접 주입은 학습 씬에서 비활성화.
- [ ] 검증: Heuristic 모드로 텔레옵했을 때 기존 직접 주입과 동일하게 손이 따라오는지 확인
  (경로 재배선이 추종 품질을 깨지 않았는지).

### 6-2. 데모 수집 조건
- [ ] **실시간(1배속)으로만 녹화** — 사람이 시연하므로 배속 불가.
- [ ] `DemonstrationRecorder`로 `.demo` 파일 기록. 에피소드 단위 = 리셋 단위.
- [ ] **성공 에피소드만 저장** (실패분 컷): 파지 성공 판정을 화면에 표시해 셀프 판정.
- [ ] 수량 가이드: GAIL 병용 기준 **성공 에피소드 30~50개**부터 시작, BC 비중을 높이려면 100+.
- [ ] **다양성**: 물체 스폰 랜덤화를 켠 채로 수집 (같은 위치만 시연하면 일반화 실패).
- [ ] 데모 파일에 스펙 버전(§5-3) 태깅 — obs/action이 바뀌면 어떤 데모가 무효인지 추적.

### 6-3. 데모 품질 게이트 (수집 후, 학습 전)
- [ ] 데모 재생(replay)으로 성공률 확인 — 기록된 action을 그대로 실행해 ≥ 80% 재현되면 합격.
  (물리 비결정성으로 100%는 안 나옴. 50% 미만이면 노이즈 과다 → §2-2로 회귀)

## 7. Phase 10 — 학습 실행 (IL → RL 순서)

### 7-1. 권장 학습 전략
스파스 보상 + 고차원 연속 행동(9~15 DoF)은 순수 RL 탐색이 거의 실패한다. 표준 레시피:
1. **BC(Behavioral Cloning) 사전학습**: config의 `behavioral_cloning` 섹션, 데모로 정책 초기화.
2. **GAIL + PPO 본학습**: `gail` reward signal(시연 흉내 보상) + `extrinsic`(태스크 보상) 병행.
   초기엔 gail 비중 크게(strength 0.5~1.0), 학습 진행되면 extrinsic 위주로.
3. 수렴 후 **GAIL 비중 축소/제거하고 PPO 파인튜닝** — 시연 수준을 넘어서게 함.

### 7-2. 학습 실행 조건
- [ ] config YAML 작성 (PPO 하이퍼파라미터 + behavioral_cloning + gail + curriculum).
- [ ] **headless 빌드**로 학습 (`--num-envs 4~8`, `-batchmode -nographics`) — 에디터 학습은 디버깅용만.
- [ ] TensorBoard 모니터링: cumulative reward, gail/extrinsic 분리 곡선, entropy.
- [ ] 평가 프로토콜: 학습 중단 기준(보상 정체 N스텝), 성공률 측정용 고정 시드 평가 씬(랜덤화 고정) 100 에피소드.

### 7-3. 반복 루프
성공률 낮음 → 원인 분해: ① 데모 품질(§6-3 재확인) ② 보상 설계(§5-2 조정) ③ 물리(§4 프로브 재실행)
④ 관측 부족(접촉 정보 추가 등). **한 번에 하나만 바꾸고 A/B** — 진동 디버깅에서 확립한 방법론 그대로.

## 9. 예상 함정 (이 프로젝트 이력 기반)

- **버전 지옥**: mediapipe 전례. mlagents는 Python 3.10 전용 — 3.11 전역 환경에 설치 시도 금지. (§3-1)
- **배속에서만 나오는 진동**: 초경량 링크+고강성 드라이브라 후보 1순위. 1배속 안정을 근거로 건너뛰지 말 것. (§4-3)
- **리셋 오염**: 초기 킥(문제 ⑭)의 재림 — 리셋 프로브 100회는 과해 보여도 RL에선 최소치. (§4-4)
- **스펙 변경으로 데모 폐기**: action 정의를 바꾸는 순간 녹화분 전량 무효. 동결 규율 유지. (§5-2)
- **포트 충돌**: udp_test_receiver 켜두면 Unity에 패킷 안 감(문제 ⑨) — 데모 녹화 세션 체크리스트에 포함.
- **에디터 학습의 착시**: 에디터에서는 되는데 headless 빌드에서 깨지는 경우(경로·해상도 의존 코드) —
  일찍 빌드로 옮겨 검증.
